# ADR-008: Correlation ID for Distributed Tracing

## Status
Accepted

## Context

The showcase application is built on an event-driven microservices architecture, where multiple services communicate asynchronously via Kafka and synchronously via REST APIs. A single logical operation, such as initiating a payment, involves interactions across several services (Payment Service, Settlement Service, Ledger Service, Reporting Service).

In such a distributed environment, it becomes challenging to:
-   Trace the end-to-end flow of a single request or event.
-   Debug issues that span multiple services.
-   Monitor the performance and behavior of interconnected components.

The `arc42.md` document identifies "Correlation ID" as a cross-cutting concept and "Observability" as a key quality goal, underscoring the necessity for a robust tracing mechanism.

## Decision

A unique `correlationId` will be generated at the entry point of every logical request (e.g., when a payment API call is received) and propagated consistently across all subsequent service calls (REST, Kafka messages) and log entries throughout the system. Additionally, domain-specific identifiers like `paymentId`, `messageId`, `endToEndId`, and `instructionId` will also be propagated to provide richer context.

## Rationale

1.  **Enhanced Observability**: Provides a clear, unambiguous way to link all related events and log messages generated by different services for a single logical operation.
2.  **Simplified Debugging and Troubleshooting**: Drastically reduces the time and effort required to diagnose issues by allowing developers to quickly filter and follow the complete execution path of a request across the entire distributed system.
3.  **Improved Monitoring**: Enables the creation of end-to-end transaction monitoring and performance metrics, helping to identify bottlenecks or failures in specific parts of the flow.
4.  **Auditability and Compliance**: Facilitates auditing by providing a complete, chronological record of all actions related to a specific transaction, which is crucial for financial systems.
5.  **Consistency**: Ensures a consistent approach to tracing across all services, regardless of the communication protocol used.

## Consequences

### Positive
-   Significantly improves the ability to understand, debug, and monitor the distributed system.
-   Facilitates faster incident response and problem resolution.
-   Provides a foundation for integrating with advanced distributed tracing tools (e.g., OpenTelemetry, Zipkin).
-   Enhances the overall reliability and maintainability of the application.

### Negative
-   Requires consistent implementation across all services and communication channels.
-   Adds a small overhead to message payloads (HTTP headers, Kafka message headers/payloads).
-   Developers must adhere to the propagation pattern, which can be a source of errors if not enforced.
-   Effective utilization requires a centralized logging system (e.g., ELK stack, Splunk) that can index and query based on `correlationId`.

## Implementation Notes

-   **Generation**: The `correlationId` will be generated as a UUID at the initial entry point of a request (e.g., in an API Gateway or the first service receiving an external request).
-   **Propagation (HTTP)**: For synchronous REST calls, the `correlationId` will be passed in a dedicated HTTP header (e.g., `X-Correlation-ID`).
-   **Propagation (Kafka)**: For asynchronous Kafka messages, the `correlationId` (and other relevant IDs) will be included in the Kafka message headers or, if headers are not suitable for the consumer, within the message payload itself.
-   **Logging**: All services will integrate the `correlationId` into their logging context (e.g., using SLF4J's MDC - Mapped Diagnostic Context) so that every log line generated during the processing of a request includes the `correlationId`.
-   **Domain-Specific IDs**: `paymentId`, `messageId`, `endToEndId`, and `instructionId` will be treated similarly to `correlationId` where applicable, providing granular context within the payment domain.
-   **Framework Support**: Leverage Spring Cloud Sleuth (or Spring Boot's built-in tracing capabilities with Micrometer) for automatic propagation of trace and span IDs, which can complement the custom `correlationId`.

## Alternatives Considered

1.  **No Explicit Correlation ID**:
    *   **Pros**: No implementation overhead.
    *   **Cons**: Extremely difficult to trace requests in a distributed system, leading to significant debugging challenges and poor observability. Unacceptable for a complex event-driven system.
2.  **Rely Solely on Application-Specific IDs**:
    *   **Pros**: Directly uses business context.
    *   **Cons**: While `paymentId` is crucial, it might not exist at the very beginning of a request, or a request might not directly map to a single payment. A generic `correlationId` provides a universal tracing mechanism for all types of requests.
3.  **Full Distributed Tracing Solution (e.g., OpenTelemetry, Zipkin) from the Start**:
    *   **Pros**: Provides advanced features like span visualization, dependency graphs, and latency analysis.
    *   **Cons**: Can introduce more complexity and setup overhead initially. While the `correlationId` mechanism lays the groundwork, a full tracing solution can be integrated later as the system matures and needs evolve.
