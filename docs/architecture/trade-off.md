# Trade-off Analysis of Architectural Decisions

This document summarizes the key architectural decisions made for the SEPA Instant Payment & ISO 20022 Reporting Showcase, highlighting the chosen approach and the alternatives considered. Each decision is documented in detail in its respective ADR file.

| ADR | Decision | Chosen | Alternatives Considered |
|:----|:---------|:-------|:------------------------|
| 001 | **Event-Driven Architecture with Kafka**: Use Kafka as the event backbone to decouple payment, settlement, ledger, and reporting services. Each service publishes domain events to Kafka topics and consumes events from other services asynchronously. | Apache Kafka as the event backbone | Synchronous REST calls; Message queue (RabbitMQ); Event store (Event Sourcing DB) |
| 002 | **Service-Level Routing for SEPA Credit Transfer vs. SEPA Instant Credit Transfer**: Add a `serviceLevel` field to the payment domain model with values `SEPA` or `INST`. The Settlement Service routes payment events to service-level-specific handlers based on this field. | Add `serviceLevel` field to payment domain model; Settlement Service routes based on this field. | Single generic handler with timing parameter; Separate microservices per service level; Detect service level from message content |
| 003 | **Generate ISO 20022 XML from Internal Domain Model**: The system will maintain a clean, internal domain model that represents the core business concepts. A dedicated ISO 20022 mapping and generation layer will be responsible for translating between this internal domain model and the external ISO 20022 XML messages. | Internal domain model with dedicated ISO 20022 mapping/generation layer. | Directly use ISO 20022 XML objects as the internal domain model; Use a generic data structure (e.g., `Map<String, Object>`) for the internal model |
| 004 | **Use Keycloak for OAuth2/JWT Authentication and Authorization**: Keycloak will be used as the Identity and Access Management (IAM) solution to provide OAuth2/JWT-based authentication and authorization for all exposed REST APIs. | Keycloak for OAuth2/JWT authentication and authorization. | Basic Authentication; API Keys; Custom Token-Based Authentication |
| 005 | **Use PostgreSQL as System of Record**: PostgreSQL will be used as the primary system of record for all persistent data within the showcase application, including payment, ledger, and audit records. | PostgreSQL as the primary system of record. | NoSQL Databases (e.g., MongoDB, Cassandra); In-memory Databases (e.g., H2 for testing); Other Relational Databases (e.g., MySQL, Oracle) |
| 006 | **Idempotency for Payment Requests**: All payment initiation API endpoints will be designed to be idempotent. Clients will be required to provide a unique `Idempotency-Key` header with each payment request. | Idempotency enforced via `Idempotency-Key` header for payment requests. | No Idempotency; Application-Specific Unique Identifiers; Client-Side Deduplication Only |
| 007 | **Immutable Audit Trail**: The system will implement an immutable audit trail for all significant payment lifecycle events. Once an audit record is created and persisted, it cannot be altered or deleted. | Immutable audit trail for all significant payment lifecycle events. | Modifiable Audit Logs; Rely Solely on Application Logs (e.g., ELK Stack); Full Event Sourcing |
| 008 | **Correlation ID for Distributed Tracing**: A unique `correlationId` will be generated at the entry point of every logical request and propagated consistently across all subsequent service calls and log entries. Domain-specific identifiers will also be propagated. | Propagation of `correlationId` and domain-specific IDs for distributed tracing. | No Explicit Correlation ID; Rely Solely on Application-Specific IDs; Full Distributed Tracing Solution (e.g., OpenTelemetry, Zipkin) from the Start |
| 009 | **Centralized Exception Handling and Retry Mechanisms**: The system will implement a centralized strategy for exception handling, incorporating retry mechanisms for transient failures and Dead Letter Queues (DLQs) for persistent failures. APIs will return standardized error responses. | Centralized exception handling with retry mechanisms, Dead Letter Queues (DLQs), and standardized error responses. | No Explicit Retry/DLQ; Manual Retry/Recovery Only; External Workflow Engine for Retries |
